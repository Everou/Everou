使用静态工厂方法代替类的构造器

使用静态工厂方法的好处：

1. 有直观的方法名，有方法名更为直观的反馈该方法的作用。
2. 可以复用对象，减少开支。比如单例模式。
3. 可以返回该类的任意子类。这是服务提供者框架的基础

静态工厂方法的缺点：

1. 当类没有非公有的或者受保护的构造方法时,静态工厂无法返回其子类（不明白，如果使用静态内部类提供工厂方法可以解决？）
2. 静态工厂方法并没有过多约定俗成的名字,它与其他的静态方法并没有什么差异

然而双重检验锁并不是完美的，因为singleton = new Singleton()这个操作并不是原子性的，它所做的又以下几件事情：

1. 堆内存中开辟空间
2. 调用Singleton类的构造方法进行初始化
3. 将内存地址的hashicode赋予引用，此操作保证singleton != null

jvm在不存在happen-before请关系情况下会进行指令重排，无法保障2,3两步的先后顺序。当线程A进入同步块中，并且进行了1,3两步操作,线程B调用了方法，此时singleton != null,直接return singleton,而此时的singleton并没有完成初始化,会抛出异常。解决方法,在1.5之后使用volatile修饰singleton,volatile的作用为:

1. 刷新内存屏障，保证可见性(实际上就是在read一定会在write的操作上)
2. 禁止指令重排

避免使用终结方法finalize()

一个对象被GC，至少经历两次标记过程：当对象进行可达性分析发现其没有与GC Root相连接的引用链，即该对象不可达。对象会被第一次标记和筛选，其筛选的依据是，对象是否有必要执行finalize()方法。当对象没有重写finalize()方法或者重写并已执行过，被视为没有必要执行finalize()方法。
当对象被判定为有必要执行finalize()方法时，对象会被放入一个叫做F-Queue的队列中，由虚拟机自身维护的一个低优先级的线程Finzlizer将会执行F-Queue中对象的finalize()方法。然而这里的执行是指会去触发这个方法，而不承诺会等该方法执行完毕。这种做法的原因是，当一个对象的finalize()方法执行及其缓慢或者发生了死循环，将导致F-Queue中的其他对象永远处于等待状态，导致整个内存的回收系统都奔溃。finalize()方法是对象被回收的最后一次救赎，当finalize()方法中让对象自身与引用链建立连接，那么对象被在第二次标记时移除“即将回收”的集合中，而如果finalize()方法并没有让其与引用链建立联系，那么第二次标记之后对象就真的被回收了。
